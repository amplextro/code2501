# Code 2501 仕様書 v0.3

## コンセプト

フォント適用するだけでUTF-8テキストを2Dバーコード化できるコード体系。
XRグラス等を通してのみ可読な「見えない文章」を実現する。
名称はバーコード規格の命名慣例（Code 128, Code 39等）に準拠。

## 命名規則

| 用途 | 名称 |
|---|---|
| 正式名 | Code 2501 |
| CLI / パッケージ名 | code2501 |
| ファイル拡張子 | .c25 |
| フォント名 | Code 2501 Mono |

## 基本パラメータ

| パラメータ | 値 |
|---|---|
| セル値 | モノクロ（1bit: 黒=1, 白=0） |
| 縦行数 | 16行（固定） |
| 横幅 | グリフごとに可変（プロポーショナル） |
| 読み取り順 | 左上→右→次行（ラスタスキャン） |
| 対応文字 | Unicode（UTF-8エンコーディング） |

## グリフ構造

### 全グリフ共通

```
列0: 同期バー（黒）
列1: 同期バー（白）
列2〜: ペイロード
```

- 左端2列は全グリフ共通の同期パターン（黒白の縦線）
- デコーダはこれを検出してグリフ境界・行位置を特定する

### グリフ種別一覧

| 種別 | 用途 | 横幅 |
|---|---|---|
| START | 開始Finder + ヘッダ情報 | 固定幅 |
| CHAR | 1文字のデータ | UTF-8バイト長に比例 |
| ECC | 誤り訂正ブロック | ECCシンボル数に比例 |
| STOP | 終端Finder + 全体チェックサム | 固定幅 |

## START グリフ

```
列0-1  : 同期バー
列2-9  : Finderパターン（8列 × 16行の固定パターン）
列10-25: ヘッダ領域
```

### Finderパターン

位置・回転・スケール検出用。上半分と下半分で異なるパターンを持ち、
上下の判別にも使用する。

```
上8行: ████████   (全黒)
       █      █
       █ ████ █
       █ ████ █
       █      █
       ████████
       (空白行)
       ▀▀▀▀▀▀▀▀   (交互パターン: クロック)

下8行: 上記の反転（上下判別用）
```

### ヘッダ領域（16列 × 16行 = 256bit）

| フィールド | ビット数 | 説明 |
|---|---|---|
| Version | 4 | 仕様バージョン（0〜15） |
| ECC Level | 2 | L(00) / M(01) / Q(10) / H(11) |
| ECC Interval | 6 | ECCグリフの挿入間隔（文字数, 1〜63） |
| Reserved | 116 | 将来拡張用 |
| Header ECC | 128 | ヘッダ自体のRS誤り訂正 |

## CHAR グリフ（文字グリフ）

### レイアウト

```
列0-1    : 同期バー
列2      : バイト長フラグ（2bit, 行0-1に配置、残り14行は同期補助）
列3〜    : データセル
末尾2列  : グリフ内パリティ（列単位XOR）
```

### データセル配置

UTF-8バイト列をラスタスキャン順に配置し、
空き行にはグリフ内Reed-Solomon ECCシンボルを充填する:

```
行0:       [bit7..bit0]  ← 1st byte (data)
行1:       [bit7..bit0]  ← 2nd byte (data, if any)
行2:       [bit7..bit0]  ← 3rd byte (data, if any)
行3:       [bit7..bit0]  ← 4th byte (data, if any)
行4-15:    [bit7..bit0]  ← per-glyph RS ECC symbols
```

- ASCII (1バイト): data 1行 + ECC 15行 → 訂正能力が最も高い
- 2バイト文字: data 2行 + ECC 14行
- 3バイト文字: data 3行 + ECC 13行
- 4バイト文字: data 4行 + ECC 12行

ECCはGF(2⁸)上のReed-Solomon符号。データバイト列に対して
(16 - バイト長) 個のECCシンボルを生成し、空き行に配置する。
これにより全グリフが16行すべてを使用し、視覚的密度が均一になる。

### バイト長フラグ

| 値 | UTF-8バイト長 | データ列数 |
|---|---|---|
| 00 | 1バイト (ASCII) | 8 |
| 01 | 2バイト | 8 |
| 10 | 3バイト | 8 |
| 11 | 4バイト | 8 |

- 横幅は全て8データ列で統一（1行に1バイト、縦方向にバイトを積む）
- ASCII文字は行0のみ使用、漢字は行0-2を使用

### グリフ内パリティ

末尾2列にデータ+ECC列のXORパリティを配置。
全16行が埋まっているため、パリティもデータ＋ECCの両方をカバーする。

### CHAR グリフ総横幅

```
同期(2) + バイト長(1) + データ(8) + パリティ(2) = 13列（全文字共通）
```

**注**: 横幅が全文字共通＝等幅フォントとなる。
START/STOPグリフは26列（CHAR 2グリフ分）で統一し、
複数行表示時のグリッド整列を保証する。

## ECC グリフ

直前N文字分のReed-Solomon GF(2⁸) 誤り訂正シンボルを格納。

### レイアウト

```
列0-1  : 同期バー
列2    : ECCマーカー（全行黒 = ECCグリフであることを示す）
列3〜  : RS符号シンボル（ラスタスキャン配置）
```

### ECCパラメータ

| ECCレベル | 訂正能力 | シンボル数（対データ比） |
|---|---|---|
| L | 約7% | 低 |
| M | 約15% | 中 |
| Q | 約25% | 高 |
| H | 約30% | 最高 |

具体的なシンボル数はECC Interval値との組み合わせで決定。
（プロトタイプ実装後に実験して最適値を決定する）

## STOP グリフ

```
列0-1  : 同期バー
列2-9  : Finderパターン（STARTと左右反転した形）
列10-25: 全体チェックサム（16列 × 16行 = 256bit）
```

STARTのFinderと左右反転にすることで、読み取り方向の判別にも使用。

## 行間（複数行レイアウト）

テキストが複数行にわたる場合、グリフ行の間にセパレータ領域を挿入する。

### 構造（計4行）

```
[████ 16行のグリフ行 ████]
[▀▄▀▄▀▄▀▄▀▄▀▄▀▄▀▄▀▄▀▄▀▄]  ← セパレータパターン（1行）
[                         ]  ← 空白行（1行）
[                         ]  ← 空白行（1行）
[▀▄▀▄▀▄▀▄▀▄▀▄▀▄▀▄▀▄▀▄▀▄]  ← セパレータパターン（1行）
[████ 16行のグリフ行 ████]
```

### セパレータパターン

- 全幅にわたる1セル交互パターン（白黒白黒...）
- デコーダが行間を能動的に検出するためのクロック信号として機能
- 画像の歪み補正時の基準点としても利用可能

### 設計根拠

- 4行 = グリフ高さ（16行）の25%
- 視覚的に十分な行間隔を確保しつつ、面積効率を維持
- 同期バー（左端の黒列）が途切れることで、グリフ行と行間の境界を判別可能
- フォント実装ではline-height等で制御可能な比率

### フォント運用での行間

フォントのline-heightをグリフ高さの1.25倍に設定し、
上下のセパレータパターンはフォントのascent/descent領域に配置する。

## 読み取りフロー

```
1. 画像からFinderパターン（START/STOP）を検出
2. 回転・スケール・歪み補正
3. 同期バーからグリフ境界を検出
4. 各グリフの種別を判定（バイト長フラグ or ECCマーカー）
5. CHARグリフからUTF-8バイト列を復元
6. ECCグリフでブロック単位の誤り訂正を実行
7. グリフ内パリティで残存エラーを検出
8. UTF-8文字列として出力
```

## フォント運用

### 使用方法

1. Code 2501フォントをインストール
2. テキストの先頭に START文字（U+E000, Private Use Area）を挿入
3. テキストの末尾に STOP文字（U+E001）を挿入
4. N文字ごとに ECC文字（U+E002）を挿入
5. フォントを適用 → 2Dコード表示

### Private Use Area マッピング

| Unicode | グリフ種別 |
|---|---|
| U+E000 | START |
| U+E001 | STOP |
| U+E002 | ECC |
| U+0000〜U+10FFFF | CHAR（通常のUnicode文字） |

### 制約事項

- ECCグリフはSTART内のヘッダに記載されたInterval値と整合する必要がある
  → v1.0ではツール側で自動挿入を提供予定
- フォント適用だけでは動的なECC計算はできないため、
  ECCグリフの中身は事前にエンコーダで計算してPUA領域にマッピングする必要がある

### ECC運用の現実的な方式

ECCグリフの中身はデータ依存のため、単一のU+E002では表現できない。
実際の運用は以下のいずれかになる:

**方式A: エンコーダ + フォント（v0.3採用）**
- エンコーダが入力テキストからSTART/ECC/STOPを含む完全な文字列を生成
- ECCシンボルはPUA領域（U+E010〜U+E0FF等）に動的マッピング
- フォント適用で表示

**方式B: 完全フォントのみ（将来検討）**
- ブロックECCを諦め、グリフ内RS ECCのみで運用
- フォント適用だけで完結し、各グリフは独立した訂正能力を持つ
- ブロック跨ぎの訂正はできないが、グリフ内ECCが強力（特にASCII: 15シンボル）
